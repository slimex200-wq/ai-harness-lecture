0-10분

문제 공감

AI가 매번 다른 답을 하는 이유와 도구별 입구 차이를 설명한다.

10-25분

하네스 구조

Skill, MCP, OMC/OMX와 AGENTS, CLAUDE, DESIGN의 역할을 보여준다.

25-40분

GitHub 운영

Issue는 작업 카드, PR은 검수 공간이라는 식으로 비개발자 언어로 설명한다.

40-60분

실습

각자 프로젝트의 상태, 확인 방법, 금지 결정을 한 장 워크시트에 적는다.

문제

채팅창 설명만으로는 프로젝트 기억이 유지되지 않는다

반복 설명

매번 배경을 다시 말하고, AI가 기억한다고 믿어야 한다.

문서로 고정

AI가 먼저 읽는 프로젝트 지도를 만들면 도구가 바뀌어도 기준은 남는다.

도구 지형도

AI를 쓰는 입구는 하나가 아니다

Web

웹 채팅

빠르게 묻고 답하기 좋지만, 내 프로젝트 파일과 실행 환경은 약하다.

App

데스크톱 앱

파일, 브라우저, 터미널을 한곳에서 다루며 작업 흐름을 오래 가져간다.

IDE

Cursor / VS Code

코드 편집과 리뷰에 강하고, 개발자 작업 흐름에 가장 가깝다.

Terminal

CLI

테스트, 빌드, git, 자동화에 강하다. 대신 맥락 문서가 더 중요하다.

오픈소스

오픈소스 도구는 “작업 방식을 빌려올 수 있다”

좋은 점

Clawd 같은 도구의 상태 표시, 훅, 테마 구조를 보고 내 작업 방식에 맞게 응용할 수 있다.

주의할 점

코드 라이선스와 이미지/캐릭터 라이선스는 다를 수 있다. 쓸 때는 범위를 확인한다.

Skill / MCP

Skill은 일하는 법, MCP는 연결하는 법이다

Skill

AI에게 특정 작업 방식을 알려주는 절차서다. 예: 코드 리뷰 방법, 발표자료 만드는 방법, 리서치 방식.

MCP

AI가 외부 도구와 데이터에 접근하게 하는 연결 규격이다. 예: GitHub, Figma, DB, 브라우저.

Skill은 행동 규칙, MCP는 도구 연결이다.

OMC / OMX

OMC/OMX는 AI 실행을 안정시키는 런타임 하네스다

상태

Memory

작업 상태, 계획, 로그를 남겨 다음 세션에서도 이어갈 수 있게 한다.

흐름

Modes

plan, team, review, verify 같은 작업 모드를 만들어 AI가 덜 즉흥적으로 움직이게 한다.

검증

Checks

테스트, 빌드, 리뷰, 완료 보고를 반복 가능한 절차로 묶는다.

핵심 구분

OMC/OMX가 있어도 프로젝트 MD는 필요하다

OMC / OMX

AI가 어떻게 움직일지 정하는 실행 엔진이다. 모드, 훅, 팀, 상태 저장을 담당한다.

AGENTS / CLAUDE / DESIGN

이 프로젝트가 무엇인지 알려주는 지도다. 제품 사실, 결정, 검증, 취향을 담당한다.

큰 그림

복붙 프롬프트 1

처음 만들 때 쓰는 프롬프트

내 프로젝트를 AI와 함께 안정적으로 작업할 수 있게 하네스 문서를 만들어줘. 먼저 README, package.json, 기존 AGENTS.md / CLAUDE.md / DESIGN.md가 있으면 읽어줘. 만들 문서: - AGENTS.md: 얇은 공통 출입문 - CLAUDE.md: Claude 전용 어댑터 - PROJECT_STATE.md / CHECKS.md / DECISIONS.md - DESIGN.md: 제품 톤과 금지 패턴 AGENTS.md에 모든 내용을 몰아넣지 말고, 전용 문서로 분리해줘. 마지막에 GitHub Issue 작업 카드 3개도 제안해줘.
복붙 프롬프트 2

이미 문서가 있을 때는 이렇게 보완한다

현재 AGENTS.md / CLAUDE.md / DESIGN.md / PROJECT_STATE.md / CHECKS.md / DECISIONS.md를 읽고 서로 충돌하거나 중복되는 내용을 찾아줘. 원칙: - AGENTS.md는 얇은 공통 출입문으로 유지 - CLAUDE.md는 AGENTS.md를 따르는 어댑터로 유지 - 실제 프로젝트 사실은 전용 문서로 이동 - DESIGN.md에는 화면 톤, 금지 패턴, 검증 기준만 남김 수정 전에는 어떤 문제가 있는지 요약하고, 수정 후에는 변경 파일과 남은 리스크를 알려줘.
